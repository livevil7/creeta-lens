# Lens 문서 작성 컨벤션 (Document Writing Conventions)

> Lens 플러그인이 생성하는 모든 문서의 작성 규칙 및 표준 정의서
> 버전: 1.0.0 | 작성일: 2026-03-02

---

## 적용 범위

이 컨벤션은 Lens가 **런타임에 생성하는 문서**에 적용된다.

| 대상 | 생성 주체 | 현재 파일 |
|------|-----------|-----------|
| 작업계획서 (Work Plan) | `/cp` | `docs/YYYY-MM-DD-{slug}.md` |
| 실행 결과 합성 | `/cc` | `.lens/results/synth-YYYY-MM-DD-{slug}.md` (saveSynthesisResults=true 시) |
| Plan 상태 추적 | `plan-manager.js` | `.lens/plan-state.json` |
| Agent 대시보드 | `agent-tracker.js` | `.lens/agent-dashboard.json` |

프로젝트 자체 문서(`CLAUDE.md`, `README.md`, `CHANGELOG.md`)의 규칙은 `docs/DOCUMENTATION-GUIDE.md`를 따른다.

## 1. 문서 네이밍 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| 파일명 | `YYYY-MM-DD-{slug}.md` | slug 생성 규칙이 `plan-manager.js`에만 있고 SKILL.md와 중복 정의됨 |
| 폴더 | `docs/` (기본), `.lens/results/` (/cc) | 작업계획서는 `docs/`, /cc 합성 결과는 `.lens/results/` |
| ID | `plan_${timestamp36}_${hex}` | Plan에만 ID 존재. Agent, Session과 ID 형식이 제각각 |

### 개선된 규칙

#### 1.1 파일명 형식

```
{type}-{YYYY-MM-DD}-{slug}.md
```

| 접두사 (type) | 문서 종류 | 생성 스킬 | 예시 |
|---------------|-----------|-----------|------|
| `plan` | 작업계획서 | `/cp` | `plan-2026-03-02-jwt-auth-login.md` |
| `result` | 실행 결과 합성 | `/cc` | `result-2026-03-02-seo-audit.md` |
| `report` | 분석/리포트 | 향후 확장용 | `report-2026-03-02-perf-analysis.md` |

> **근거**: 타입 접두사가 있으면 `ls` 한 번으로 종류별 필터링이 가능하다.
> 현재 `YYYY-MM-DD-slug.md` 형식은 날짜순 정렬에는 유리하지만, 작업계획서와 결과 문서가 섞이면 구분이 안 된다.

#### 1.2 Slug 생성 규칙 (단일 원천: `plan-manager.js`)

```javascript
// 규칙: plan-manager.js의 generateSlug()이 유일한 slug 생성 함수
// SKILL.md에서는 "slug는 plan-manager.js가 생성한다"고만 명시

// 허용 문자: 모든 Unicode 문자(\p{L}), 숫자(\p{N}), 하이픈
// 최대 길이: 50자
// 단어 수: 3~5개 핵심 키워드
// 1글자 단어: 제외 (filter(w => w.length > 1))
// 폴백: slug가 비어있거나 2자 미만이면 plan-{random_hex_6}
```

#### 1.3 폴더 구조

```
project-root/
├── docs/                        # /cp 작업계획서 (기본 저장 위치, planDir로 변경 가능)
│   ├── 2026-03-02-jwt-auth.md
│   └── 2026-03-01-db-migration.md
├── .lens/
│   ├── results/                 # /cc 합성 결과 (saveSynthesisResults=true 시)
│   │   └── synth-2026-03-02-seo-audit.md
│   ├── plan-state.json          # Plan 상태 추적
│   └── agent-dashboard.json     # Agent 라이프사이클
```

#### 1.4 ID 형식 통일

| 엔티티 | 현재 형식 | 통일 형식 |
|--------|-----------|-----------|
| Plan | `plan_${ts36}_${hex4}` | `plan_${ts36}_${hex4}` (유지) |
| Session | `sess_${ts36}_${hex6}` | `sess_${ts36}_${hex4}` (hex 길이 통일) |
| Agent | `agent_${ts36}_${hex4}` | `agt_${ts36}_${hex4}` (접두사 축약) |

```
공통 패턴: {prefix}_{base36_timestamp}_{hex_random_4}
예시: plan_m1abc_f3d2, sess_m1abc_a1b2, agt_m1abc_c4e5
```

## 2. 문서 헤더/메타데이터 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| 헤더 | `# Work Plan / 작업계획서` + blockquote 메타 | 파싱 불가. 메타데이터가 자유 텍스트 |
| Plan ID | blockquote 안에 인라인 표기 | 프로그래밍적으로 추출 불가 |
| 날짜 | `Generated by Lens Plan on {YYYY-MM-DD HH:mm}` | 형식만 있고 타임존 규칙 없음 |

### 개선된 규칙

#### 2.1 YAML 프론트매터 (필수)

모든 Lens 생성 문서는 YAML 프론트매터로 시작한다.

```yaml
---
id: plan_m1abc_f3d2
type: plan                    # plan | result | report
version: 1                   # 문서 스키마 버전 (수정할 때마다 증가 아님, 포맷 변경 시만)
created: 2026-03-02T14:30:00+09:00   # ISO 8601 + 타임존
updated: 2026-03-02T15:00:00+09:00   # 최종 수정 시각
status: draft                 # draft | approved | executing | completed | failed | cancelled
generator: /cp                # 생성한 스킬
language: ko                  # 본문 언어 (ISO 639-1)
tags: [jwt, auth, backend]    # 검색/필터용 태그
---
```

#### 2.2 필수 vs 선택 필드

| 필드 | 필수 | 타입 | 설명 |
|------|------|------|------|
| `id` | 필수 | string | 고유 식별자 |
| `type` | 필수 | enum | 문서 종류 |
| `version` | 필수 | integer | 프론트매터 스키마 버전 (현재 `1`) |
| `created` | 필수 | ISO 8601 | 생성 시각 (타임존 포함) |
| `updated` | 필수 | ISO 8601 | 최종 수정 시각 |
| `status` | 필수 | enum | 문서/작업 상태 |
| `generator` | 필수 | string | 생성 스킬명 |
| `language` | 필수 | string | ISO 639-1 언어 코드 |
| `tags` | 선택 | string[] | 태그 목록 |
| `parent_id` | 선택 | string | 연결된 상위 문서 ID |
| `related` | 선택 | string[] | 관련 문서 ID 목록 |

#### 2.3 `plan-state.json` 스키마와의 정합성

프론트매터의 `status`와 `plan-state.json`의 `status`는 **반드시 동기화**한다.
상태 변경 시 두 곳 모두 업데이트:

```
1. plan-state.json의 해당 plan ID 엔트리의 status 변경
2. plan 마크다운 파일의 프론트매터 status 변경
→ 둘 다 plan-manager.js가 책임진다
```

## 3. 다국어 처리 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| 언어 감지 | "Respond in the user's language" (SKILL.md) | 구체적 감지 로직 없음 |
| 헤더 | `## Task / 요청사항` (이중 언어 고정) | 사용자가 영어면 한국어 헤더 불필요 |
| 본문 | "user's language" | 기본값 규칙 없음 |
| 설정 | `defaultPlanLanguage: null` (lens.config.json) | 사용 방법 미정의 |

### 개선된 규칙

#### 3.1 언어 결정 우선순위

```
1. lens.config.json의 defaultPlanLanguage (명시적 설정) → "ko", "en", 등
2. 사용자 입력 언어 자동 감지 (요청 텍스트 기반)
3. 폴백: "en" (영어)
```

#### 3.2 헤더 이중 언어 규칙

| 본문 언어 | 헤더 형식 | 예시 |
|-----------|-----------|------|
| `en` | 영어만 | `## Task` |
| `ko` | 영어 / 한국어 | `## Task / 요청사항` |
| `ja` | 영어 / 일본어 | `## Task / 要件` |
| `zh` | 영어 / 중국어 | `## Task / 任务` |
| 기타 | 영어 / 해당 언어 | `## Task / Tarea` |

**규칙**: 영어 헤더는 항상 첫 번째. 비영어 언어일 때만 ` / {번역}` 추가.

#### 3.3 본문 언어 규칙

| 요소 | 언어 | 이유 |
|------|------|------|
| 프론트매터 키 | 영어 | YAML 키는 코드. 다국어화 불가 |
| 섹션 헤더 | 이중 언어 (3.2 규칙) | 가독성 + 검색성 |
| 본문 텍스트 | 사용자 언어 | 사용자가 읽는 주요 콘텐츠 |
| 코드/명령어 | 영어 | 코드는 원본 유지 |
| 상태 값 | 영어 (enum) | `draft`, `completed` 등 파싱 가능해야 함 |

#### 3.4 언어 코드 표 (지원 언어)

| 코드 | 언어 | Lens 지원 상태 |
|------|------|-----------------|
| `en` | English | 기본값 |
| `ko` | 한국어 | 지원 |
| `ja` | 日本語 | 지원 |
| `zh` | 中文 | 지원 |
| `es` | Español | 지원 |
| `fr` | Français | 지원 |
| `de` | Deutsch | 지원 |
| `it` | Italiano | 지원 |

## 4. 상태 관리 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| Plan 상태 | `Draft` → (실행) → `Completed \| Partial \| Failed` | `Approved`, `Executing` 단계 누락 |
| 상태 위치 | 마크다운 본문 하단 `**Status**: Draft` | 프론트매터와 이원화. 파싱 어려움 |
| `/cc` 결과 | 상태 개념 없음 | 성공/실패 이력 없음 |
| Agent | `pending → running → done \| error` | Plan 상태와 연결 안 됨 |

### 개선된 규칙

#### 4.1 문서 상태 머신

```
                     ┌──────────┐
                     │  draft   │ ← 최초 생성
                     └────┬─────┘
                          │ 사용자 승인
                     ┌────▼─────┐
                     │ approved │
                     └────┬─────┘
                          │ 실행 시작
                   ┌──────▼──────┐
                   │  executing  │
                   └──┬───────┬──┘
                      │       │
               ┌──────▼──┐ ┌──▼───────┐
               │completed│ │  failed  │
               └─────────┘ └──────────┘

   취소는 어디서든 가능:
   draft ──→ cancelled
   approved ──→ cancelled
```

#### 4.2 상태 전이 규칙

| 현재 상태 | 허용 전이 | 전이 조건 |
|-----------|-----------|-----------|
| `draft` | `approved`, `cancelled` | 사용자 응답 |
| `approved` | `executing`, `cancelled` | 실행 시작 또는 취소 |
| `executing` | `completed`, `failed` | 모든 스킬 완료 또는 에러 |
| `completed` | (최종 상태) | - |
| `failed` | (최종 상태) | - |
| `cancelled` | (최종 상태) | - |

#### 4.3 상태 변경 시 필수 기록

| 전이 | 기록 사항 | 기록 위치 |
|------|-----------|-----------|
| → `approved` | `approvedAt` 타임스탬프 | 프론트매터 `updated` + `plan-state.json` |
| → `executing` | 실행 시작 시각 | 프론트매터 `updated` |
| → `completed` | `completedAt` + 결과 요약 | 프론트매터 `updated` + 본문 Execution Result 섹션 |
| → `failed` | `failedAt` + 실패 원인 | 프론트매터 `updated` + 본문 Execution Result 섹션 |
| → `cancelled` | `cancelledAt` + 사유 | 프론트매터 `updated` + `plan-state.json` |

#### 4.4 `/cc` 결과 문서의 상태

`/cc`는 계획 없이 즉시 실행하므로 상태 머신이 단순하다:

```
executing → completed | failed
```

`result` 타입 문서의 프론트매터에 `status: completed` 또는 `status: failed`로 기록.

## 5. 문서 연결 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| Plan → Result | Plan 파일 하단에 Execution Result 섹션 추가 | 한 파일에 계획과 결과가 혼재. 파일이 길어짐 |
| Result → Plan | 없음 | 역참조 불가 |
| Plan → Plan | 없음 | 연속 작업의 흐름 추적 불가 |
| `/cc` → `/cp` | 없음 | `/cc` 결과를 바탕으로 `/cp` 계획을 세울 때 연결 없음 |

### 개선된 규칙

#### 5.1 연결 방식: 프론트매터 참조

```yaml
---
id: plan_m1abc_f3d2
parent_id: plan_m0xyz_a1b2       # 이 문서의 선행 문서
related: [result_m1abc_e3f4]     # 관련 문서 목록
---
```

#### 5.2 연결 시나리오

| 시나리오 | 연결 방법 |
|----------|-----------|
| Plan 실행 후 결과 기록 | Plan 파일 하단에 `## Execution Result` 섹션 추가 (현재 방식 유지). 결과가 복잡하면 별도 `result` 문서 생성 후 `related`로 연결 |
| 연속 Plan | 후속 Plan의 `parent_id`에 선행 Plan ID 기재 |
| `/cc` 결과 → `/cp` 계획 | Plan의 `parent_id`에 Result 문서 ID 기재 |
| 실패 후 재시도 | 새 Plan의 `parent_id`에 실패한 Plan ID 기재 |

#### 5.3 본문 내 참조 형식

문서 본문에서 다른 문서를 참조할 때:

```markdown
> 이전 계획 [plan_m0xyz_a1b2](../plans/plan-2026-03-01-db-migration.md) 에서 이어지는 작업입니다.
```

형식: `[{document_id}]({relative_path})`

#### 5.4 Execution Result 섹션 (Plan 파일 내)

Plan 파일에 결과를 인라인으로 추가할 때의 표준 형식:

```markdown
---

## Execution Result / 실행 결과

> Completed on {YYYY-MM-DDTHH:mm:ss+TZ}

### Outcomes / 실행 결과 상세
- [ ] {기대 결과 1} → {실제 결과}
- [x] {기대 결과 2} → {실제 결과}

### Files Changed / 변경된 파일
| File | Action | Description |
|------|--------|-------------|
| `src/auth.js` | Modified | JWT 검증 로직 추가 |
| `src/middleware.js` | Created | 인증 미들웨어 신규 생성 |

### Status: {Completed | Failed}
```

> **기대 결과 항목**: Plan 단계의 `## Expected Outcomes`의 체크리스트와 1:1 매핑. 달성한 항목은 `[x]`, 미달성은 `[ ]`.

## 6. 품질 기준 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| 필수 섹션 | SKILL.md에 템플릿으로만 정의 | "이 섹션이 없으면 거부" 같은 규칙 없음 |
| 구체성 | 없음 | "JWT 구현" vs "Express.js 4.x에서 RS256 JWT 발급/검증 구현" 구분 안 됨 |
| 검증 | 없음 | AI가 생성한 문서가 품질 기준을 충족하는지 확인 방법 없음 |

### 개선된 규칙

#### 6.1 작업계획서 (Plan) 필수 섹션

| # | 섹션 | 필수 여부 | 최소 요구사항 |
|---|------|-----------|---------------|
| 1 | 프론트매터 | 필수 | 2.1의 모든 필수 필드 |
| 2 | Task / 요청사항 | 필수 | 사용자 원문 + 1줄 이상 컨텍스트 |
| 3 | Matched Skills | 필수 | 1개 이상 매칭된 스킬. Why Selected 칼럼 비어있으면 안 됨 |
| 4 | Execution Plan | 필수 | Step별 Skill + Input + Expected output. 최소 1 Step |
| 5 | Expected Outcomes | 필수 | 체크리스트 형태. 최소 1개. 검증 가능한 구체적 결과물 |
| 6 | Risks & Considerations | 선택 | 복잡한 작업(3 Step 이상)이면 필수 |
| 7 | Execution Mode | 필수 | Single 또는 Parallel 명시 |

#### 6.2 구체성 수준 기준

**나쁜 예** vs **좋은 예**:

| 항목 | 나쁜 예 (거부) | 좋은 예 (승인) |
|------|---------------|---------------|
| Task | "인증 구현" | "Next.js 14 App Router 프로젝트에 Google OAuth 2.0 로그인 추가" |
| Expected Output | "코드 생성" | "`src/app/api/auth/[...nextauth]/route.ts` 파일 생성, Google provider 설정 포함" |
| Input | "사용자 요청" | "사용자의 원본 요청 + `package.json`의 현재 의존성 목록" |
| Risk | "문제가 생길 수 있음" | "기존 session 기반 인증과 충돌 가능 → 마이그레이션 가이드 포함 필요" |

#### 6.3 검증 체크리스트 (AI 셀프 체크)

Plan 생성 후, 사용자에게 제시하기 전에 AI가 내부적으로 확인:

```
[ ] 프론트매터의 필수 필드 7개 모두 존재하는가?
[ ] 모든 Matched Skill이 실제 세션에 존재하는 스킬인가?
[ ] Expected Outcomes가 검증 가능한 형태인가? (파일명, 동작, 수치 등)
[ ] Execution Plan의 각 Step에 Input과 Expected Output이 있는가?
[ ] 본문 언어가 3.1 우선순위 규칙을 따르는가?
[ ] 상태가 "draft"로 설정되어 있는가?
```

#### 6.4 실행 결과 (Result) 품질 기준

| # | 항목 | 필수 여부 | 기준 |
|---|------|-----------|------|
| 1 | 완료 타임스탬프 | 필수 | ISO 8601 |
| 2 | Expected Outcomes 매핑 | 필수 | Plan의 체크리스트와 1:1 대응 |
| 3 | 변경 파일 목록 | 필수 | 파일이 변경된 경우 |
| 4 | 최종 상태 | 필수 | Completed / Failed |
| 5 | 실패 원인 | Failed일 때 필수 | 에러 메시지 또는 원인 설명 |

## 7. 템플릿 분리 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| Plan 템플릿 | `skills/cp/SKILL.md` 안에 마크다운 코드 블록으로 하드코딩 | 템플릿만 수정하려면 SKILL.md 전체를 편집해야 함. 워크플로우 로직과 템플릿이 혼재 |
| `/cc` 합성 포맷 | `skills/cc/SKILL.md` 안에 ASCII art 박스로 하드코딩 | 동일 문제 |
| 확장 | 불가능 | 사용자가 자신만의 템플릿을 정의할 방법 없음 |

### 개선된 규칙

#### 7.1 템플릿 파일 분리 원칙

```
유지: SKILL.md에는 워크플로우 로직만 기술
분리: 템플릿은 별도 파일로 추출
참조: SKILL.md에서 템플릿 파일 경로를 명시
```

#### 7.2 템플릿 파일 위치

```
lens/
├── templates/                    # (신규) 템플릿 파일 디렉토리
│   ├── plan.md                   # /cp 작업계획서 템플릿
│   ├── result.md                 # /cc 실행 결과 합성 템플릿
│   └── execution-result.md       # Plan 파일 하단에 추가하는 결과 섹션 템플릿
├── skills/
│   ├── cp/SKILL.md               # 워크플로우만 (템플릿 → templates/plan.md 참조)
│   └── cc/SKILL.md               # 워크플로우만 (템플릿 → templates/result.md 참조)
```

#### 7.3 SKILL.md에서의 참조 방식

```markdown
### 3. Generate Plan Document

Create a work plan document using the template at `templates/plan.md`.
Read the template file using the Read tool, fill in all `{placeholder}` values,
and save the result to `.lens/plans/` with the standard file naming convention.
```

#### 7.4 템플릿 변수 규칙

템플릿 내 변수는 중괄호 `{variable_name}` 형식:

| 변수 | 타입 | 설명 |
|------|------|------|
| `{id}` | string | 문서 ID (generatePlanId() 결과) |
| `{created}` | ISO 8601 | 생성 시각 |
| `{status}` | enum | 초기 상태 (`draft`) |
| `{language}` | string | ISO 639-1 코드 |
| `{task}` | string | 사용자 원문 요청 |
| `{skill_table}` | markdown table | 매칭된 스킬 테이블 |
| `{execution_steps}` | markdown | Step별 실행 계획 |
| `{expected_outcomes}` | markdown checklist | 기대 결과 체크리스트 |
| `{risks}` | markdown list | 위험 요소 (선택) |
| `{execution_mode}` | string | Single 또는 Parallel |

#### 7.5 사용자 커스텀 템플릿 (향후)

`lens.config.json`에서 커스텀 템플릿 경로 지정:

```json
{
  "templates": {
    "plan": "./my-templates/custom-plan.md",
    "result": null
  }
}
```

- `null`이면 기본 템플릿 사용
- 지정된 경로의 파일이 없으면 기본 템플릿으로 폴백

## 8. 에러/실패 기록 컨벤션

### 현재 상태

| 항목 | 현재 규칙 | 문제점 |
|------|-----------|--------|
| 스킬 실행 실패 | `/cc`에서 "note it briefly and continue" | 기록 형식, 위치 미정의 |
| Agent 에러 | `agent-tracker.js`에 `error` 필드 (최대 500자) | 에이전트 단위 에러만. 문서 단위 에러 없음 |
| Plan 실패 | `Status: Failed`만 기록 | 실패 원인, 재시도 가이드 없음 |
| 부분 성공 | `Status: Partial` | "어디까지 성공했고 어디서 실패했는지" 구조화 안 됨 |

### 개선된 규칙

#### 8.1 에러 기록 3계층

```
Level 1: Agent 에러     → agent-dashboard.json (자동, 훅이 기록)
Level 2: 문서 에러      → Plan/Result 파일 본문 (구조화된 섹션)
Level 3: 세션 에러 요약  → plan-state.json의 status + failedAt
```

#### 8.2 실패 시 Plan 파일에 기록할 내용

```markdown
---

## Execution Result / 실행 결과

> Failed on {YYYY-MM-DDTHH:mm:ss+TZ}

### Outcomes / 실행 결과 상세
- [x] {성공한 기대 결과} → {실제 결과}
- [ ] {실패한 기대 결과} → **FAILED**: {실패 원인}

### Error Details / 에러 상세

| Step | Skill | Status | Error |
|------|-------|--------|-------|
| 1 | /skill-a | Completed | - |
| 2 | /skill-b | **Failed** | `TypeError: Cannot read property 'x' of undefined` |
| 3 | /skill-c | Skipped | Step 2 실패로 미실행 |

### Recovery Suggestion / 복구 제안
- {구체적인 복구 방법}
- {대안 접근법}

### Status: Failed
```

#### 8.3 부분 성공 처리

- Plan에 3개 Step이 있고, Step 1-2는 성공, Step 3은 실패한 경우:
  - **Status**: `failed` (전체 Plan은 실패)
  - **각 Step 상태**: 개별적으로 Completed/Failed/Skipped 표기
  - **Recovery Suggestion**: 실패한 Step부터 재시도하는 방법 기재

> **`Partial` 상태 폐지**: 기존 `Partial`은 모호하다. 대신 `failed` 상태 + 본문에서 성공/실패 Step을 명시한다.

#### 8.4 `/cc` 병렬 실행 실패

```markdown
### Agent Execution Summary / 에이전트 실행 요약

| # | Skill | Status | Duration | Error |
|---|-------|--------|----------|-------|
| 1 | /skill-a | Done | 3.2s | - |
| 2 | /skill-b | Error | 1.5s | Timeout after 30s |
| 3 | /skill-c | Done | 5.1s | - |

### Partial Results Available
{성공한 에이전트들의 결과는 정상 표시. 실패한 에이전트는 "Error" 표기 후 이유 명시}
```

#### 8.5 에러 메시지 형식

```
에러 메시지는 다음 형식을 따른다:
- 첫 줄: 에러 유형 (TypeError, NetworkError, TimeoutError 등)
- 둘째 줄: 발생 위치 (파일명 + 함수명 또는 스킬명)
- 셋째 줄: 원인 요약 (자연어)
- 최대 500자
```
