## 기술적 접근

### Pre-mortem 병렬 실행 전략 (Phase B의 핵심)

**왜 Claude만 / Codex만이 아닌가**: 이미 합의된 결론 — 두 모델이 같은 리스크를 지적하면 "진짜 심각", 한 쪽만 지적하면 "각 모델 강점 부각". 사용자는 병기 결과를 보고 직접 판단.

**구현 단면**: `/cp` PLAN 모드는 현재 선형 (Phase 1→2→3→4). Phase 2.5를 삽입할 때 **Phase 2에서 먼저 파일을 저장**하고, Phase 2.5에서 파일을 **다시 읽어 `## ⚠️ 사전 리스크` 섹션만 업데이트**하는 2-write 패턴이 가장 안전. Phase 2.5 실패해도 계획 문서는 이미 존재.

**병렬 실행 방법**: Claude Code의 단일 메시지 multi-tool-call 패턴 사용. Opus pre-mortem은 내부 추론(현재 세션 모델이 opus이므로 Task tool 불필요, 동일 컨텍스트에서 수행), Codex 호출은 Bash tool로 `codex exec --skip-git-repo-check "..."` 실행. 둘 다 tool call로 낸 뒤 결과 수렴.

**대안 (탈락)**: Task tool로 Opus 전용 agent spawn → 세션 컨텍스트 handoff 비용 발생, 이득 없음.

### Supervisor 조건부 opus 승격 (Phase C의 논리)

**현재 모순**: Worker (Hard) = opus인데 Supervisor = sonnet. 시니어 산출물을 주니어가 채점하는 구조.

**수정 원칙**: Supervisor 모델 ≥ max(Worker models). 즉 Worker에 opus 하나라도 있으면 Supervisor도 opus.

**비용 고려**: sonnet → opus 승격은 단일 Supervisor 호출이라 전체 태스크 비용 상승폭 작음 (~10~15%). 반면 버그/설계 누락 시 재작업 비용이 훨씬 큼.

**대안 (탈락)**: "항상 opus" → 단순 태스크에도 과잉 비용. 조건부가 최적.

### Skill 강제 할당 (Phase D의 한계와 타협)

**Claude Code 아키텍처 제약**: Task tool의 subagent는 스킬 호출을 강제할 수 없음. 유일한 수단은 **프롬프트 명령문 + 완료 보고 검증**.

**핵심 전술**:
1. Worker 프롬프트에 "첫 액션 = Skill invoke" 명시 (압박)
2. 완료 보고 필수 형식 `Skill invoked: /X` (추적)
3. Supervisor가 이 라인 누락 확인 시 자동 재작업 지시 (강제)

**실패 확률**: 여전히 Worker가 무시할 수 있음. 그러나 Supervisor 감사로 loop가 재시작되므로 결국 Skill 호출하게 됨. 이는 **인센티브 설계로 규범 강제** 방식.

**대안 1 (부분 탈락)**: `subagent_type`으로 전문 agent (`pr-review-toolkit:code-reviewer` 등) spawn — v3.2로 미룸. 이유: 매핑 테이블 유지보수 부담, agent-to-skill 매칭이 아직 완벽하지 않음.

**대안 2 (탈락)**: Hook으로 Worker Task tool 호출 시 스킬 invoke 주입 — Claude Code hooks는 Task 호출 자체를 intercept할 수 있지만 payload 수정은 불안정.

### Codex CLI 호출 규약

**표준 호출**: `codex exec --skip-git-repo-check "$PROMPT"` (비대화형 1회 응답, git repo 체크 스킵)

**출력 파싱**: Codex 출력에는 `reasoning summaries: none`, `session id: ...`, `--------`, `user`, `codex`, `tokens used N,NNN` 같은 메타 라인이 섞임. 본문만 추출하려면:
1. `codex` 라벨 다음 줄부터
2. `tokens used` 라벨 이전 줄까지

정규식 또는 awk로 슬라이스. 파싱 로직은 `docs/rules/codex-integration.md` 에 코드 포함.

**미설치 감지**: 3단계 순차 시도
1. `command -v codex` (PATH)
2. `ls /c/Users/ADMIN/.vscode/extensions/openai.chatgpt-*/bin/windows-x86_64/codex.exe` (VSCode 확장)
3. 실패 → `codex_available=false` 플래그로 pre-mortem 단독 모드

**에러 처리**: Codex 호출 중 실패 (timeout, 인증 만료) 시 exception 잡아서 pre-mortem 문서에 "Codex 호출 실패: {에러 요약}" 기록하고 Opus 결과만 사용.

### Worker 보고 형식 표준화

**현재 형식**: 자유 서술. Supervisor가 파싱하기 어려움.

## 주니어가 놓칠 통찰

- **Phase 2.5는 Phase 2와 원자화되지 않음**: pre-mortem 실패해도 계획 문서는 저장돼야 함. Phase 2 완료 = 저장, Phase 2.5 = 섹션 업데이트. 두 작업을 분리해야 복구 가능.
- **Worker 스킬 강제 시 순환 위험**: Worker가 스킬 호출 거부 → Supervisor fail → 재작업 → 같은 패턴 반복. 최대 반복 5회 제한(기존 Lens 규칙)이 이미 있으므로 5회 후 사용자 알림으로 탈출.
- **Codex 응답의 Non-determinism**: Codex는 같은 프롬프트에도 다른 리스크 뽑음. Pre-mortem 결과가 매번 달라도 "최소 1회는 잡아냈다" 가치가 있음 — 결정적 검증이 아니라 샘플링.
- **Supervisor opus 자기인식**: "당신의 모델은 opus"라고 명시하지 않으면 모델이 스스로 보수적으로 행동. 명시해야 opus 특유의 깊은 추론 발현.
- **Codex CLI는 `--skip-git-repo-check` 필수**: git repo 아닌 임시 디렉토리에서 호출하면 기본적으로 "이 디렉토리 git 아님" 에러. pre-mortem은 계획 문서 내용만 보내므로 repo 체크 불필요.
